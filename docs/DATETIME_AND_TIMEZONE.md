# Время и часовые пояса

## Как время связано с БД

- **В БД (PostgreSQL/Prisma)** все даты хранятся в **UTC** (`timestamp with time zone`). Это стандарт: при сохранении и чтении мы работаем с UTC.
- **На фронте** пользователь видит даты в **своём часовом поясе и языке** (определяются автоматически и из профиля).

## Определение часового пояса и языка

| Что | Источник |
|-----|----------|
| Часовой пояс (IANA) | `Intl.DateTimeFormat().resolvedOptions().timeZone`, сохраняется в `User.timezone` |
| Смещение устройства (для API) | `-new Date().getTimezoneOffset()` — минуты впереди UTC, не зависит от VPN |
| Язык (локаль) | Telegram: `initDataUnsafe.user.language_code`, иначе `navigator.language` |

## VPN и «время на устройстве»

При использовании VPN **IANA timezone** (`Intl.resolvedOptions().timeZone`) может совпадать с зоной VPN-сервера, а не с реальным местом пользователя.

Поэтому для **фильтра по дню** мы опираемся на **смещение устройства** (device offset):

- На клиенте передаётся **`timezoneOffsetMinutes`** — минуты относительно UTC из `-new Date().getTimezoneOffset()`. Это значение дают **системные часы устройства**, а не геолокация или VPN.
- Дата для запроса считается по **локальной дате на устройстве** (`getFullYear()`, `getMonth()`, `getDate()` у `Date`), т.е. «какой день видит пользователь».
- Сервер при наличии `timezoneOffsetMinutes` строит диапазон UTC по этому смещению и по переданной дате. Так «сегодня» и выбранный день совпадают с тем, что пользователь видит на экране, даже при включённом VPN.

**Нагрузка на сервер:** передаётся одно число (минуты) в query. Никакого «считывания времени с устройства» на сервере нет — всё делается на клиенте, сервер только принимает параметры. Доп. нагрузки нет.

## Трекер и фильтр по дате (параметры API)

1. Пользователь выбирает дату — на клиенте она превращается в **локальную дату устройства** (`YYYY-MM-DD`).
2. Запрос: `GET /tasks/:userId?date=2026-03-20&timezoneOffsetMinutes=180` (и при необходимости `&timezone=Europe/Moscow` для fallback).
3. Сервер при наличии **`timezoneOffsetMinutes`** (приоритет) строит UTC-диапазон для этого календарного дня по смещению. Иначе используется **`timezone`** (IANA), затем по умолчанию — UTC.
4. Отображение дат в интерфейсе — по `locale` и `timezone` (для подписей и формата).

## Создание задач и даты

- При создании задачи с «Due date» фронт отправляет дату в формате **ISO 8601** (например, `2026-03-20` или `2026-03-20T14:30:00.000Z`).
- Бэкенд сохраняет в БД как UTC. Для отображения и фильтра в трекере снова используется смещение/часовой пояс и локаль пользователя.
